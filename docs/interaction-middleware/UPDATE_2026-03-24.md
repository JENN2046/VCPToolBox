# Interaction Middleware Update 2026-03-27

This note records the current implementation state so the docs match the codebase.

## What changed

### AdapterContract Standardization (2026-03-27)

- Created `AdapterContract` base class with standardized interface for all platform adapters
- Implemented `DingTalkAdapter` with full contract compliance (decodeInbound, encodeOutbound, sendBySession)
- Implemented `WecomAdapter` with full contract compliance
- Implemented `OneBotAdapter` with full contract compliance
- Updated `ChannelHubService` to:
  - Dynamically load adapter instances from contract.js on first use
  - Use `decodeInbound()` for inbound event processing
  - Use `verifySignature()` for platform signature validation
  - Use `sendBySession()` for outbound delivery

### Previous Updates (2026-03-24)

- `ChannelHub` outbound delivery is no longer a fake success path.
- `DeliveryJob` now carries delivery target context such as `session`, `target`, and `metadata`.
- `AdapterRegistry` now supports runtime adapter instance registration and lookup.
- `CapabilityRegistry` now resolves `qq -> onebot` and `feishu -> lark` aliases.
- `messageProcessor.js` now recursively resolves `SarPrompt` and `Tar/Var` file contents through the full variable resolver.
- A regression test file was added for `messageProcessor.js` coverage.

## Current status

- Core ChannelHub control plane is fully implemented.
- **AdapterContract** is standardized with:
  - `authenticate(requestContext)` - adapter-level authentication
  - `verifySignature(requestContext)` - platform signature verification
  - `decodeInbound(rawPayload, context)` - convert platform payload to Event
  - `encodeOutbound(reply, context)` - convert NormalizedReply to platform messages
  - `sendBySession(sessionDescriptor, platformMessages, context)` - send to target session
  - `healthCheck()` - return adapter health status
- Three adapters fully implemented: DingTalk, WeCom, OneBot
- Adapters are loaded dynamically at runtime (no startup registration needed)
