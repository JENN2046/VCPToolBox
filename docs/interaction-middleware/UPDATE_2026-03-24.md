# Interaction Middleware Update 2026-03-24

This note records the current implementation state so the docs match the codebase.

## What changed

- `ChannelHub` outbound delivery is no longer a fake success path.
- `DeliveryJob` now carries delivery target context such as `session`, `target`, and `metadata`.
- `AdapterRegistry` now supports runtime adapter instance registration and lookup.
- `CapabilityRegistry` now resolves `qq -> onebot` and `feishu -> lark` aliases.
- `messageProcessor.js` now recursively resolves `SarPrompt` and `Tar/Var` file contents through the full variable resolver.
- A regression test file was added for `messageProcessor.js` coverage.

## Current status

- Core ChannelHub control plane is largely complete.
- Outbound closure still depends on runtime adapter instances being registered at startup.
- Message processor recursion is stabilized, but dynamic fold behavior still depends on plugin-provided RAG interfaces.

## Next steps

1. Register concrete adapter runtime instances during adapter startup.
2. Add a small adapter runtime contract document for `sendBySession` / `deliver` / `sendMessage`.
3. Add more end-to-end tests around outbound delivery once a runnable adapter instance is available in the test environment.
